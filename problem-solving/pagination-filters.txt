pattern to implement filters and pagination in angular with rest api

Here's a clean, scalable pattern for implementing filters, pagination, and sorting in Angular (standalone components recommended in Angular 17+) with a REST API using server-side processing.This is the most common and performant approach for real-world applications.1. Recommended ArchitectureData Service: Handles all HTTP calls and builds query parameters.
Component: Manages UI state (filters, page, sort) and subscribes to data.
Model: Define interfaces for request/response.
Optional: Use Angular Material Table (mat-table) + MatPaginator + MatSort for rich UI.

2. Define Models

// models/pagination.model.ts
export interface PaginationParams {
  page: number;           // 0-based or 1-based (depends on your API)
  pageSize: number;
  sortField?: string;
  sortDirection?: 'asc' | 'desc';
  filters?: Record<string, any>;  // e.g. { status: 'active', search: 'john' }
}

export interface PaginatedResponse<T> {
  data: T[];
  total: number;          // total records (for paginator)
  page?: number;
  pageSize?: number;
  // add other metadata if your API returns it
}

3. Data Service (Best Practice)


// services/data.service.ts
import { Injectable } from '@angular/core';
import { HttpClient, HttpParams } from '@angular/common/http';
import { Observable } from 'rxjs';
import { PaginationParams, PaginatedResponse } from '../models/pagination.model';

@Injectable({ providedIn: 'root' })
export class DataService {
  private apiUrl = 'https://api.example.com/items';   // your endpoint

  constructor(private http: HttpClient) {}

  getData(params: PaginationParams): Observable<PaginatedResponse<any>> {
    let httpParams = new HttpParams()
      .set('page', params.page.toString())
      .set('pageSize', params.pageSize.toString());

    // Sorting
    if (params.sortField) {
      httpParams = httpParams.set('sort', params.sortField);
      if (params.sortDirection) {
        httpParams = httpParams.set('order', params.sortDirection);
      }
    }

    // Filters (flexible)
    if (params.filters) {
      Object.keys(params.filters).forEach(key => {
        const value = params.filters![key];
        if (value !== null && value !== undefined && value !== '') {
          httpParams = httpParams.set(key, value.toString());
        }
      });
    }

    return this.http.get<PaginatedResponse<any>>(this.apiUrl, { params: httpParams });
  }
}

API Parameter Examples (common conventions):Pagination: ?page=1&pageSize=10 or ?page=0&limit=10
Sorting: ?sort=createdAt&order=desc
Filters: ?status=active&search=john&minPrice=100

Adapt the parameter names to match your backend.

4. Component Implementation (Standalone)

// components/data-table.component.ts
import { Component, OnInit, ViewChild, signal } from '@angular/core';
import { MatPaginator, PageEvent } from '@angular/material/paginator';
import { MatSort, Sort } from '@angular/material/sort';
import { DataService } from '../../services/data.service';
import { PaginatedResponse } from '../../models/pagination.model';
import { debounceTime, Subject } from 'rxjs';

@Component({
  selector: 'app-data-table',
  standalone: true,
  imports: [ /* MatTableModule, MatPaginatorModule, etc. */ ],
  templateUrl: './data-table.component.html'
})
export class DataTableComponent implements OnInit {
  data = signal<any[]>([]);
  totalRecords = signal(0);
  pageSize = signal(10);
  pageIndex = signal(0);
  isLoading = signal(false);

  filters: Record<string, any> = {};
  private filterSubject = new Subject<void>();

  @ViewChild(MatPaginator) paginator!: MatPaginator;
  @ViewChild(MatSort) sort!: MatSort;

  displayedColumns = ['id', 'name', 'status', 'createdAt'];

  constructor(private dataService: DataService) {
    // Debounce filter changes
    this.filterSubject.pipe(debounceTime(400)).subscribe(() => this.loadData());
  }

  ngOnInit() {
    this.loadData();
  }

  loadData() {
    this.isLoading.set(true);

    const params = {
      page: this.pageIndex(),
      pageSize: this.pageSize(),
      sortField: this.sort?.active,
      sortDirection: this.sort?.direction as 'asc' | 'desc' | undefined,
      filters: this.filters
    };

    this.dataService.getData(params).subscribe({
      next: (res: PaginatedResponse<any>) => {
        this.data.set(res.data);
        this.totalRecords.set(res.total);
        this.isLoading.set(false);
      },
      error: (err) => {
        console.error(err);
        this.isLoading.set(false);
      }
    });
  }

  onPageChange(event: PageEvent) {
    this.pageIndex.set(event.pageIndex);
    this.pageSize.set(event.pageSize);
    this.loadData();
  }

  onSortChange(sort: Sort) {
    this.loadData();
  }

  applyFilter(key: string, value: any) {
    this.filters[key] = value;
    this.pageIndex.set(0);        // reset to first page on filter
    this.filterSubject.next();    // debounce
  }

  clearFilters() {
    this.filters = {};
    this.pageIndex.set(0);
    this.loadData();
  }
}

I am using pagination, sorting and filters.  When user enters input in a filter I should send the first page?

Yes, that's the correct and standard behavior.When the user changes any filter (search input, dropdown, date, etc.), you should always reset to the first page (pageIndex = 0 or page = 1 depending on your API).Why you should reset to first page on filter change:Filters change the result set → the current page (e.g., page 5) may no longer make sense.
It gives the best user experience — user sees the most relevant results from the top.
This is how almost all professional apps work (Google, Amazon, GitHub, admin panels, etc.).

// When any filter changes
applyFilter(key: string, value: any) {
  if (value === '' || value === null || value === undefined) {
    delete this.filters[key];   // clean up empty filters
  } else {
    this.filters[key] = value;
  }

  this.resetToFirstPage();
  this.loadData();
}

private resetToFirstPage() {
  this.pageIndex.set(0);
  
  // Sync with Material Paginator
  if (this.paginator) {
    this.paginator.pageIndex = 0;
  }
}

// For sorting (optional - you can also reset on sort)
onSortChange() {
  this.resetToFirstPage();   // or keep current page - depends on UX preference
  this.loadData();
}

Explain debouncing for filters

Debouncing is a very important technique when working with filters (especially search inputs) in Angular + REST API applications.What is Debouncing?Debouncing delays the execution of a function (in your case, calling the API) until the user stops typing for a certain amount of time.Real-world analogy:Imagine someone is typing a long search term like "angular pagination filters".Without debouncing → API call fires on every keystroke: a → an → ang → angu... → 20+ unnecessary API calls.
With debouncing (300ms) → API call fires only once, 300 milliseconds after the user stops typing.

This gives a much better user experience and drastically reduces server load.Why Use Debouncing for Filters?Scenario
Without Debounce
With Debounce (300ms)
User types quickly

1–2 API calls
Network usage & server load
Very high
Very low
Performance
Slow / laggy
Smooth
Cost (if API is paid)
High
Low

Best debounce delay for search filters: 300ms to 400ms is the sweet spot.How to Implement Debouncing in Angular (Modern Way)Here’s the cleanest pattern using RxJS (still the most common and reliable approach):

import { Component, OnInit, signal } from '@angular/core';
import { Subject, debounceTime } from 'rxjs';
import { DataService } from './data.service';

@Component({
  selector: 'app-data-table',
  standalone: true,
  templateUrl: './data-table.component.html'
})
export class DataTableComponent implements OnInit {

  data = signal<any[]>([]);
  totalRecords = signal(0);
  isLoading = signal(false);

  filters: Record<string, any> = {};

  // Subject to trigger debounced API call
  private filterSubject = new Subject<void>();

  constructor(private dataService: DataService) {
    // Debounce logic
    this.filterSubject
      .pipe(debounceTime(350))        // Wait 350ms after last change
      .subscribe(() => {
        this.resetToFirstPage();
        this.loadData();
      });
  }

  ngOnInit() {
    this.loadData();   // initial load
  }

  // Called from template on every keystroke
  onFilterChange(key: string, value: any) {
    if (value === '' || value == null) {
      delete this.filters[key];
    } else {
      this.filters[key] = value;
    }

    this.filterSubject.next();   // Trigger debounced action
  }

  private resetToFirstPage() {
    this.pageIndex.set(0);
    if (this.paginator) this.paginator.pageIndex = 0;
  }

  loadData() { ... }   // your existing API call method
}

Explain throttle vs debounce

Here's a clear and practical explanation of Throttle vs Debounce — especially in the context of Angular filters, search inputs, and API calls with pagination.Simple AnalogyDebounce → "Wait until the user stops doing something, then act once."Like a friend who only replies after you finish talking.

Throttle → "Act at a maximum rate (e.g., once every 300ms), no matter how fast the user acts."Like a friend who answers you at most once every few seconds, even if you keep talking.

Key DifferencesFeature
Debounce
Throttle
When it executes
After a quiet period (no new events)
At regular intervals (leading, trailing, or both)
Best for
Events that should happen after user stops
Events that should happen during continuous activity
Number of executions
Usually 1 (the last one)
Multiple, but limited (e.g., once every 300ms)
Typical delay behavior
Resets timer on every event
Ignores events in between fixed intervals
API calls for search
Very few (only after typing stops)
More frequent (steady updates while typing)

Visual Example (Typing "Angular Filters")User types: A → n → g → u → l → a → r (quickly)No control: 7 API calls
Debounce (350ms): Only 1 API call after the user stops typing for 350ms
Throttle (350ms): Several API calls (e.g., at start, then every 350ms while typing)

When to Use Which in Angular Filters & PaginationUse Debounce (Recommended for your case):Search inputs / text filters
Any filter where the final value matters (user is still typing)
You want to avoid sending many unnecessary requests to the REST API

This is why we used debounceTime(350) in the previous example for the search box.Use Throttle when:Infinite scrolling (load more data while scrolling)
Mouse move or resize events
Real-time progress updates
Button spam protection (e.g., "Save" button clicked rapidly)
You want immediate feedback at a controlled rate
